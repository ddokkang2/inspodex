# Inspodex 개편 방향서

## 1. 결론

현재 Inspodex는 "레퍼런스 탐색 런처"로는 충분히 유용하다. 하지만 "직관적이고도 풍성한 데이터를 활용하는 레퍼런스 검색 사이트의 표본"이 되려면, 카드 목록 중심 구조에서 벗어나 **통합 검색형 레퍼런스 데이터베이스**로 올라가야 한다.

핵심 전환은 세 가지다.

1. 디렉터리 중심 탐색에서 통합 검색 중심 탐색으로 전환
2. 얕은 카드 데이터에서 풍부한 메타데이터 중심 구조로 전환
3. 단일 스크립트 중심 구현에서 모듈형 구조와 데이터 파이프라인으로 전환

## 2. 현재 상태 진단

실제 렌더 테스트 기준으로 확인한 현재 상태는 아래와 같다.

- 장점
  - 디렉터리 폭이 넓다. 디자인 `199`, 캐릭터 `200`, 사진 `103`, 팔레트 `200`, 포즈 `260`으로 총 `962`개 수준이다.
  - 디렉터리별 빠른 필터가 잘 설계되어 있다.
  - 카드 밀도와 탐색 속도가 빠르다.
  - 포즈/팔레트는 현재 구조 안에서 가장 완성도가 높다.
  - 정적 사이트라 배포와 유지가 단순하다.
- 한계
  - 상단 조작 패널 비중이 커서 첫인상이 "영감 탐색"보다 "폼 조작"에 가깝다.
  - 모바일에서는 컨트롤이 첫 화면을 과도하게 점유한다.
  - 검색이 디렉터리별로 분리되어 있어 교차 탐색이 안 된다.
  - 데이터 깊이가 부족하다. 많은 항목이 사실상 `id`, `ko`, `en`, 얕은 `tags/q` 수준이다.
  - 상세 정보, 출처, 유사 항목, 컬렉션, 저장, 비교 기능이 없다.
  - 외부 검색 연결이 핵심 가치라 내부 정보 탐색의 체류 가치가 낮다.
  - 로직이 [`script.js`](/e:/@vscode/Inspodex/script.js)에 집중돼 개편 리스크가 크다.

## 3. 목표 제품 정의

개편 후 Inspodex의 목표는 아래로 재정의한다.

> 스타일 이름을 모아둔 링크 디렉터리가 아니라, 시각 레퍼런스를 구조적으로 탐색하고 조합할 수 있는 검색형 데이터베이스

사용자가 해야 하는 대표 작업은 다음 네 가지다.

1. 특정 무드나 목적에 맞는 레퍼런스를 빠르게 찾는다.
2. 다른 매체의 유사 레퍼런스를 교차 탐색한다.
3. 선택한 레퍼런스를 바로 작업 재료로 쓴다.
4. 여러 레퍼런스를 저장하고 조합해 무드보드처럼 활용한다.

## 4. 제품 원칙

- 검색은 디렉터리보다 위에 있어야 한다.
- 모든 레퍼런스는 최소한의 설명 가능한 메타데이터를 가져야 한다.
- 카드 자체보다 선택 이후의 정보 확장이 더 중요하다.
- 외부 검색은 보조 기능이어야 하며, 제품의 핵심 경험을 대체하면 안 된다.
- 모바일은 축소판이 아니라 별도 탐색 흐름으로 설계해야 한다.

## 5. 정보 구조 개편

### 5.1 탐색 구조

현재의 `디자인 / 캐릭터 / 사진 / 팔레트 / 포즈`는 유지하되, "데이터 사일로"가 아니라 "뷰 필터"로 바꾼다.

- 최상단에는 통합 검색창 1개를 둔다.
- 디렉터리 버튼은 검색 결과를 좁히는 상위 타입 필터로 바꾼다.
- 첫 화면은 단순 카드 그리드보다 아래 구성이 낫다.
  - 통합 검색
  - 추천 컬렉션
  - 최근 인기 태그 또는 탐색 시작점
  - 타입별 대표 카드 섹션

### 5.2 상세 경험

현재는 카드 클릭 후 외부 검색 쿼리를 만드는 흐름이 중심이다. 이를 바꿔야 한다.

카드 클릭 시 우선 열려야 할 것은 외부 이동이 아니라 **상세 패널 또는 상세 페이지**다.

상세에는 최소한 아래 정보가 들어간다.

- 한글/영문 이름
- 1문장 요약
- 핵심 속성 5개 이상
- 관련 태그
- 비슷한 레퍼런스
- 다른 타입의 연관 레퍼런스
- 작업용 액션
  - 검색어 복사
  - 프롬프트 복사
  - 팔레트 복사
  - 외부 사이트 열기

### 5.3 저장 구조

표본급 사이트로 가려면 "찾기"만으로는 부족하다. 최소한 다음 중 하나는 있어야 한다.

- 즐겨찾기
- 컬렉션 보드
- 최근 본 항목

추천 순서는 `최근 본 항목 -> 즐겨찾기 -> 컬렉션 보드`다.

## 6. 데이터 모델 개편

현재는 [`script.js`](/e:/@vscode/Inspodex/script.js) `enrichStyle()`에서 부족한 필드를 런타임에 보완한다. 이 방향은 한계가 명확하다. 개편 후에는 데이터 자체를 풍부하게 만들어야 한다.

### 6.1 공통 스키마

모든 항목은 공통으로 아래 필드를 가진다.

```ts
type ReferenceItem = {
  id: string;
  type: 'design' | 'character' | 'photo' | 'palette' | 'pose';
  title: { ko: string; en: string };
  aliases: string[];
  summary: string;
  tags: string[];
  searchTokens: string[];
  moods: string[];
  useCases: string[];
  eras: string[];
  regions: string[];
  media: string[];
  relatedIds: string[];
  sourceLinks: { label: string; url: string }[];
  asset: {
    thumb?: string;
    fallbackMode?: 'design' | 'character' | 'photo' | 'palette' | 'pose';
  };
  actions: {
    searchQuery?: string;
    promptTemplate?: string;
  };
};
```

### 6.2 타입별 확장 필드

- `design`
  - layout
  - typography
  - contrast
  - interactionTone
- `character`
  - rendering
  - genre
  - silhouette
  - audienceTone
- `photo`
  - lightingSetup
  - lensFeel
  - shotType
  - grading
- `palette`
  - colors
  - category
  - presetLabel
  - contrastProfile
- `pose`
  - poseType
  - variant
  - bodyFocus
  - energyLevel

### 6.3 분류 체계

태그를 자동 파생만 하는 구조는 중장기적으로 약하다. 아래 taxonomy를 명시적으로 관리해야 한다.

- 타입
- 무드
- 시대
- 지역/문화
- 매체/렌더링 방식
- 용도
- 난이도
- 유사도 그룹

## 7. 검색 경험 개편

### 7.1 검색 방식

검색은 다음 3층으로 설계한다.

1. 자유 검색어
2. 상위 타입 필터
3. 공통 facet 필터

공통 facet 예시:

- 무드
- 용도
- 시대
- 색감
- 명도/대비
- 현실감
- 장르

### 7.2 결과 랭킹

현재처럼 문자열 포함 여부 중심이 아니라, 다음 가중치를 두는 편이 낫다.

- 제목 exact match
- aliases match
- searchTokens match
- tags match
- summary match
- 사용자가 최근 선택한 타입과의 연관성

### 7.3 교차 추천

현재 구조에서 가장 아쉬운 지점이다. 타입 간 연결이 거의 없다.

예시:

- `neo-brutalism-web`을 보면
  - 관련 팔레트
  - 관련 사진 무드
  - 관련 캐릭터 스타일
  - 관련 포즈
  - 관련 검색 키워드
  를 함께 보여줘야 한다.

## 8. UX 개편 방향

### 8.1 홈

현재 홈은 검색 패널 중심이다. 개편 후 홈은 탐색 시작점 중심으로 바꾼다.

- 큰 통합 검색창
- 추천 컬렉션
- 인기 탐색 축
- 타입별 대표 섹션
- 최근 본 항목

### 8.2 필터 패널

현재 패널은 기능은 많지만 밀도가 높다.

개편 방향:

- 기본은 최소 필터만 노출
- 고급 필터는 접힘 영역 또는 drawer로 이동
- 모바일에서는 하단 sheet로 분리

### 8.3 카드

현재 카드는 정보 밀도는 높지만 역할이 좁다.

카드에는 최소한 아래 역할이 들어가야 한다.

- 빠른 스캔
- 저장
- 상세 진입
- 즉시 작업 액션

카드 개선안:

- 제목, 짧은 부제, 핵심 속성 2개
- 현재 태그 3개는 유지 가능
- hover 또는 tap 시 quick actions 노출
- 선택 상태가 외부 검색어 주입보다 상세 진입 중심이 되도록 변경

### 8.4 모바일

모바일은 지금보다 별도 흐름이 강해야 한다.

- 검색창과 상위 필터만 first viewport에 남긴다.
- 세부 필터는 `필터 열기` 버튼으로 분리한다.
- 상세는 전체 화면 sheet 또는 page push로 연다.

## 9. 기술 구조 개편

### 9.1 권장 방향

대개편을 할 예정이라면 정적 배포는 유지하되, 개발 구조는 빌드 기반으로 옮기는 편이 낫다.

권장 스택:

- `Vite`
- `TypeScript`
- 정적 JSON 데이터
- 검색 인덱스 생성 스크립트
- 배포는 기존처럼 정적 호스팅

이유:

- 현재 [`script.js`](/e:/@vscode/Inspodex/script.js) 단일 파일 구조는 변경 비용이 너무 높다.
- 데이터 스키마 확장과 검증이 필요하다.
- 통합 검색과 상세 페이지를 붙이려면 모듈 구조가 필요하다.

### 9.2 모듈 분리 기준

- `data/`
  - raw datasets
  - generated index
- `lib/search/`
  - tokenizer
  - ranking
  - related item resolver
- `lib/domain/`
  - schema
  - taxonomy
  - normalizers
- `ui/`
  - search bar
  - filters
  - card grid
  - detail panel
  - collection board

### 9.3 유지할 것과 버릴 것

- 유지
  - 정적 호스팅 가능 구조
  - 빠른 카드 렌더
  - 디렉터리별 전문 필터 아이디어
  - 프롬프트/HEX 복사 액션
- 버릴 것
  - 런타임 파생에 과도하게 의존하는 데이터 품질
  - 외부 검색 주도형 메인 흐름
  - 단일 거대 스크립트

## 10. 단계별 실행안

### Phase 1. Foundation

목표: 데이터베이스형 제품으로 갈 수 있는 기반 확보

- 공통 스키마 도입
- 데이터 파일을 JSON 또는 TS 모듈로 재정리
- 통합 검색 인덱스 도입
- 상세 패널 추가
- 기존 디렉터리를 통합 검색 위의 타입 필터로 재정의

### Phase 2. Core Product

목표: "표본급 레퍼런스 검색 사이트"의 핵심 경험 완성

- 홈 개편
- 교차 추천 시스템 추가
- 저장/최근 본 항목 추가
- 모바일 필터/상세 흐름 재설계
- 카드와 상세 정보의 메타데이터 강화

### Phase 3. Power Features

목표: 작업 도구로서의 차별화

- 컬렉션 보드
- 비교 보기
- 프롬프트 빌더
- 검색 히스토리
- 사용자별 작업 세션 복원

## 11. 성공 기준

개편 성공은 아래 수준을 만족해야 한다.

- 사용자가 타입을 몰라도 통합 검색으로 원하는 레퍼런스를 찾을 수 있다.
- 각 항목은 최소 5개 이상의 의미 있는 메타데이터를 가진다.
- 상세 패널에서 외부 사이트로 나가지 않고도 항목 가치를 이해할 수 있다.
- 관련 항목 추천이 같은 타입과 다른 타입을 모두 연결한다.
- 모바일 첫 화면에서 결과 카드가 바로 보인다.
- 코드 구조상 검색, 데이터, UI가 분리되어 개별 수정이 가능하다.

## 12. 권장 결정

지금은 부분 수정보다 **리뉴얼 전제의 구조 재설계**가 맞다.

다만 한 번에 전면 교체하기보다, 아래 순서를 권장한다.

1. 데이터 스키마와 통합 검색 구조를 먼저 설계
2. 현재 UI 위에 상세 패널과 통합 검색을 얹는 중간 버전 구현
3. 그 다음 홈, 카드, 모바일 탐색 구조를 갈아엎는 순서로 진행

이 순서를 따르면 제품 방향성과 기술 리스크를 동시에 관리할 수 있다.
